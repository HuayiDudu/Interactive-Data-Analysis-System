# 项目开发规范
# 【负责人】项目负责人

## 一、Git 协作规范

### 1.1 分支策略

```
main              ← 稳定版本，仅通过 PR 合入
├── dev           ← 开发主分支，每日集成
├── feat/数据管理  ← 数据管理模块功能分支
├── feat/数据清洗  ← 数据清洗模块功能分支
├── feat/可视化    ← 可视化模块功能分支
├── feat/分析功能  ← 分析功能模块功能分支
└── feat/Web界面   ← Web 界面模块功能分支
```

- 各模块开发人员从 `dev` 创建自己的功能分支，命名规则：`feat/模块名-功能描述`
- 完成开发后发起 PR → `dev`，由 项目负责人 或另一模块开发人员 Review
- 禁止直接向 `main` 推送代码

### 1.2 Git Hooks 拦截规则

项目已配置 Git hooks（`githooks/` 目录），每次 commit 自动触发检查：

**启用方式（每人必做）：**
```bash
git config core.hooksPath githooks/
```

**pre-commit 检查（提交前，语法错误拦截提交）：**

| 检查项 | 类型 | 行为 |
|--------|------|------|
| Python 语法错误 | `.py` 文件 `compile()` 检查 | 拦截，必须修复 |
| JavaScript 语法错误 | `.js` 文件 `node --check` 检查 | 拦截，必须修复 |
| `raise NotImplementedError` | grep 匹配 | 警告，不拦截 |
| `print()` 调试语句 | grep 匹配 | 警告，不拦截 |

**commit-msg 检查（提交信息格式）：**

格式必须为 `<type>: <描述>`，type 必须是以下之一，否则拦截：
`feat` / `fix` / `docs` / `refactor` / `style` / `test` / `chore`

### 1.3 Commit 规范

```
<type>: <简短描述>

示例:
feat: 实现 CSV 文件上传解析功能
fix: 修复 IQR 异常检测中空列报错
docs: 更新数据清洗模块开发文档
refactor: 重构导出路由，提取公共方法
style: 调整代码格式，删除多余空行
```

**类型说明**：

| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复 Bug |
| docs | 文档变更 |
| refactor | 代码重构 |
| style | 格式调整（无功能变化） |
| test | 测试相关 |
| chore | 构建/工具链变动 |

### 1.4 禁止操作

- ❌ 禁止直接向 `main` 推送
- ❌ 禁止 `--force` 推送（除非 项目负责人 明确授权）

## 二、编码规范

### 2.1 Python 规范

- **Python 版本**: 3.10+
- **缩进**: 4 空格（禁用 Tab）
- **行宽**: 100 字符
- **命名**:
  - 类名: `PascalCase`（如 `CleanService`）
  - 函数/方法: `snake_case`（如 `handle_missing`）
  - 变量: `snake_case`（如 `dataset_ref`）
  - 常量: `UPPER_SNAKE_CASE`（如 `ALLOWED_EXTENSIONS`）
- **类型注解**: 函数参数和返回值必须加类型注解
- **Import 顺序**: 标准库 → 第三方库 → 本地模块，每组之间空一行

### 2.2 JavaScript 规范

- **缩进**: 4 空格
- **命名**:
  - 函数: `camelCase`（如 `handleUpload`）
  - 变量: `camelCase`（如 `currentDatasetId`）
  - 常量: `UPPER_SNAKE_CASE`（如 `API_BASE_URL`）
- **分号**: 必须使用
- **async/await**: 优先于 Promise.then()

### 2.3 HTML/CSS 规范

- **HTML**: 语义化标签，缩进 4 空格
- **CSS**: 类名使用 `kebab-case`（如 `.preview-table`）
- **ID 命名**: 模块前缀 + 功能名（如 `btn-upload`, `plot-container`）

## 三、模块开发约束

### 3.1 层间依赖规则

```
表示层 ──→ 控制层 ──→ 业务层 ──→ 数据访问层 ──→ 存储层
  (HTTP)    (函数调用) (Repository接口)  (文件/DB)
```

- **表示层**：仅调用控制层 HTTP API，禁止直接操作后端
- **控制层**：仅调用 Service 层的实例方法，禁止操作文件系统、pandas、Repository
- **业务层**：仅调用 Repository 抽象接口，禁止操作 Flask request/session
- **数据访问层**：仅操作存储层，不包含任何业务规则

### 3.2 文件归属

每个源文件标有 `【负责人】` 标记，非负责人修改前需获得对方同意。

### 3.3 接口兼容

- 各模块暴露给其他模块的接口（函数签名、HTTP 响应格式）变更前，需通知相关方
- HTTP API 响应格式：成功 `{"status": "success", "data": {...}}`，失败 `{"status": "error", "message": "..."}`

## 四、代码质量

### 4.1 异常处理

- 所有 Route 函数必须 try/except 捕获 `ValueError` 和 `Exception`
- Service 层抛出自定义异常（`ValueError` 等），Route 层转换为 JSON 错误响应
- 禁止捕获异常后 `pass` 静默忽略

### 4.2 注释规范

- 公共方法必须写 docstring（参数、返回值、异常）
- 仅代码意图不明确时才写行内注释
- 不要写"显而易见"的注释（如 `x += 1  # x 加 1`）

### 4.3 测试

- MVP 阶段：每次提交前手动测试核心流程（上传 → 清洗 → 可视化 → 分析 → 导出）
- 扩展阶段：使用 pytest 编写单元测试

## 五、沟通协作

### 5.1 接口协商

- 前端与后端模块之间的接口（HTTP API）应先确定再各自实现
- JS 模块间函数名和参数由 Web 界面模块与各模块开发者协商

### 5.2 进度同步

- 各模块完成开发后，通知 项目负责人 进行集成测试
- 遇到阻塞性问题（依赖未就绪、接口变更），及时在群内同步

### 5.3 Mock 策略

- 当依赖的模块未完成时，可使用 Mock 数据独立开发和调试
- Mock 函数应加上 `// MOCK` 注释，集成时替换为真实实现
